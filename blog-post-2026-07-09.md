# I Built an Open Source Contribution Toolkit in 57 Tokens — Here's How a Beginner Can Start Contributing Today

I'm a beginner programmer. I know what loops do, I can read Python, and I've done basic git pushes and pulls. But I'd never submitted a pull request. I didn't even know what a PR was a few hours ago.

Now I have a fully tested, spec-driven Python toolkit that helps me (and anyone else) find open-source projects to contribute to, evaluate them, fork them, make changes, validate quality, and submit pull requests — all guided by an AI development environment called Kiro.

This post is about how I built it, what it does, the problems we hit along the way, and why I think this matters for other beginners who want to break into open source but don't know where to start.

## Why I Made This

The open-source world felt inaccessible to me. Every guide I found assumed you already knew git workflows, understood project conventions, and could navigate unfamiliar codebases. The advice was always "just find a good first issue and submit a PR" — as if that's simple when you've never done it.

I wanted a system that would:
- **Find repos for me** that match my interests (math, statistics) and are small enough to understand
- **Evaluate difficulty** so I don't waste time on something too complex
- **Walk me through the mechanics** — forking, branching, committing, PR submission
- **Validate my work** before I submit so maintainers don't think I'm a "vibe coder" submitting garbage
- **Document the journey** so others can follow the same path

## What I Built

The toolkit has 13 Python modules covering the full contribution lifecycle:

**Discovery & Evaluation:**
- A scoring algorithm that ranks repos by activity, size, issue count, and contribution guidelines
- Filtering logic that finds repos with "good first issue" labels or keyword-matching issues
- Difficulty classification (easy/medium/hard) based on dependencies, lines of code, and native extensions
- License detection for 7 OSI-approved licenses
- CONTRIBUTING.md summarization (≤200 words)

**Contribution Mechanics:**
- Branch name generator (`fix/add-input-validation` format, ≤50 chars)
- Conventional commit message generator and validator
- PR title and description generator with structured sections
- Pre-submission validation engine (code style, tests, commit format, branch cleanliness, scope checking)
- Linter config detection (black, ruff, flake8, pylint)

**Progression & Documentation:**
- Difficulty progression recommender (stays at same level until 2 completions, then promotes)
- Blog post generator (this very post was seeded by it)
- Reusable Kiro steering document generator

## How I Made It — The Process

Here's the honest truth: **Kiro did the coding.** My job was making decisions.

### Step 1: I Described What I Wanted

I told Kiro: "I want to learn how to contribute to open-source Python projects. I'm a beginner. I like math and statistics. I want it to take less than 30 minutes per contribution. I want to document it as a blog post."

### Step 2: Kiro Asked Me 6 Questions

Things like: What's your Python skill level? What interests you most? Have you ever done a PR before? How much time do you want to spend? These answers shaped everything that followed.

### Step 3: Spec-Driven Development

This is where Kiro's approach really shines. Instead of jumping straight to code, we built:

1. **Requirements document** — 11 formal requirements with acceptance criteria written in EARS format (WHEN/THEN/SHALL patterns). Each requirement has a user story and testable criteria.

2. **Technical design** — 9 logical modules, 19 correctness properties, data models, a composite scoring formula, error handling strategies, and a testing plan.

3. **Task list** — 46 tasks organized in dependency waves, executed in parallel where possible.

The whole spec lives in `.kiro/specs/open-source-contribution-workflow/` and serves as living documentation of every design decision.

### Step 4: Parallel Execution

Kiro executed tasks in waves — up to 5 at a time when they had no dependencies on each other. The 13 implementation modules were built in about 3 waves, followed by property tests, then the live workflow.

### Step 5: Property-Based Testing

This is probably the most rigorous part. Instead of just writing "does this work with these 3 examples?" tests, we wrote **property-based tests** using Hypothesis that verify universal truths:

- "For ANY valid contribution type and description, the branch name is always ≤50 characters and matches the pattern"
- "For ANY set of repo metadata, the scoring function returns results sorted non-increasing and capped at 10"
- "For ANY combination of validation check results, all_passed is True if and only if every individual check passes"

50+ tests, all passing. This isn't "it worked when I tried it" — it's "it works for 100 randomly generated inputs per property."

## Issues We Hit

It wasn't perfectly smooth. Here's what went sideways:

### 1. GitHub CLI Installation

The `gh` CLI wasn't installed on my machine. Kiro installed it for me using `winget`, but the terminal couldn't find it afterward because the PATH hadn't refreshed. Lesson: after installing CLI tools, restart your terminal.

### 2. Authentication Confusion

After installing `gh`, Kiro's terminal sessions couldn't see the auth token because I authenticated in a separate terminal window. The keyring stores credentials per-user, so it worked fine once I confirmed from the right session — but it was a few minutes of "wait, why isn't this working?"

### 3. Finding Good Repos Is Hard

The first search for math/statistics repos returned mostly homework projects and personal notebooks with zero open issues. We had to try multiple search strategies (topic search, keyword search, filtering by open issues, looking for "good first issue" labels) before finding repos that actually wanted contributors.

### 4. Most Small Repos Don't Need Help

This was humbling. Very small repos (under 500 lines) often don't have issues because they're complete. The sweet spot ended up being mid-size repos (1000-5000 lines) with active maintainers who've explicitly labeled issues for newcomers.

## The Contributions — What I Actually Submitted

Here's where the rubber meets the road. Using the toolkit, I found [lockwo/distreqx](https://github.com/lockwo/distreqx) — a JAX probability distributions library with 47 stars, an Apache-2.0 license, and multiple issues explicitly labeled for newcomers.

### Contribution #1: Documentation (Easy)

**PR [#99](https://github.com/lockwo/distreqx/pull/99)** — Fleshed out the docs homepage.

The project had a "good first issue" (#22) asking for help with their rendered documentation. The homepage was minimal — just a quick example and a brief description. I added:
- A "Why distreqx?" section explaining key advantages
- Feature tables listing all available distributions and bijectors
- Additional code examples for bijectors, transformed distributions, and JAX transformations

This was pure documentation — zero risk of breaking anything, genuinely helpful for newcomers to the library.

### Contribution #2: Type Hints (Easy)

**PR [#100](https://github.com/lockwo/distreqx/pull/100)** — Added array shape dimensions to the Normal distribution's type hints.

Issue #35 asked for `jaxtyping` shape annotations. The existing code used bare `Array` everywhere without specifying shapes. I added `Float[Array, " *shape"]` annotations to all methods in the Normal distribution class — making the API clearer for IDE autocompletion and enabling more precise runtime checking.

### Contribution #3: Tests (Medium — Level Up!)

**PR [#101](https://github.com/lockwo/distreqx/pull/101)** — Added 17 new tests for `jit`, `grad`, correctness, and edge cases.

This was my first medium-difficulty contribution. Issue #15 ("Improve tests", labeled "help wanted") pointed out that tests didn't match the CONTRIBUTING.md requirements for three categories: correctness, compatibility, and edge cases. I added:
- 4 `jit` compatibility tests (sample, log_prob, cdf, entropy all work under JIT)
- 4 `grad` tests (verified gradients match analytic solutions)
- 5 correctness tests (entropy of standard normal, CDF at mean = 0.5, etc.)
- 4 edge case tests (large values, small/large scale, float64 dtype)

This one felt different. It wasn't just documentation — I was verifying actual mathematical properties of the code. And it passed.

## My Advice for Other Beginners

1. **You don't need to be good at coding.** You need to be good at following a process. AI tools handle the mechanics.
2. **Start with documentation.** It's low-risk, maintainers love it, and it forces you to read the code without having to modify behavior.
3. **Single-line changes are valid.** Adding a missing type hint, fixing an error message, adding a docstring — these are real contributions.
4. **The first PR is the hardest.** Not because the code is hard, but because the process is unfamiliar. After the first one, it becomes routine.
5. **Use tools that handle the scary parts.** Git commands, PR formatting, conventional commits — let AI deal with those while you focus on understanding what needs to change.
